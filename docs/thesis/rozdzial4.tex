\chapter{Architektura systemu}
\label{cha:architekturaSystemu}

Napisać coś o tym, że całość architektury zawiera wiele integracji, używa kilka protokołów komunikacyjnych itp.
Dobra architektura systemu powinna zapewnić łatwe skalowanie oraz utrzymanie. Należy zrozumieć wymagania zarówno funkcjonalne czyli jakie zadania system ma spełniać, oraz niefunkcjonalne czyli wydajność lub bezpieczeństwo. Innym ważnym aspektem jest podział odpowiedzialności czyli na jakie komponenty powinien zostać podzielony system.

Zdecydowano, że autonomiczny system kompletacji zamówień musi zostać podzielony na trzy aplikacje: aplikacja internetowa, aplikacja serwerowa oraz aplikacja sterowania robotem. Taka architektura zapewniła odpowiedni podział odpowiedzialności na warstwę prezentacji, warstwę logiki biznesowej i warstwę zarządzania i sterowaniem robotem. Dzięki temu wszystkie trzy systemy można rozwijać niezależnie. Dodatkowo taki podział umożliwia wykorzystanie wielu robotów, poprzez odpowiednią integrację z aplikacją serwerową. Podział na trzy aplikacje można zobaczyć na rysunku \ref{fig:c4Modelc1}, który przedstawia poziom C1 w modelu C4 architektury systemu.

Może coś o procesie decyzyjnym, jakie inne zalety ma to rozwiązanie, albo jakie wady lub ograniczenia. Zaletą jest to że aplikacja frontendowa nie komunikuje się bezpośrednio z robotem.

\subsection{Komunikacja aplikacji internetowej i serwerowej}
\label{sec:komunikacjaBackendFrontend}

\subsubsection*{REST API}

Główną odpowiedzialnością aplikacji internetowej jest integracja z użytkownikiem, który może wyświetlić listę produktów, stworzyć zamówienie czy wyświetlić historię zamówień. Do realizacji takiej komunikacji wykorzystano REST API.

API (z ang. Application Programming Interface) to zestaw reguł, które umożliwiają systemom komunikować się między sobą. REST jest stylem architektonicznym definiującym zasady projektowania systemów sieciowych, a REST API jest konkretnym interfejsem realizującym te zasady. W REST wszystko jest zasobem, a każdy zasób ma swój adres URL. Do wykonania operacji na tych zasobach wykorzystywany jest protokół HTTP.

REST API jest dobrym wyborem dla operacji inicjowanych przez użytkownika, które polegają na wysłaniu żądania i otrzymaniu odpowiedzi, szczególnie gdy nie jest wymagana komunikacja w czasie rzeczywistym. Architektura REST jest bezstanowa, co sprawia że każde żądanie jest niezależne. To sprawia, że kod po stronie serwera staje się prostszy, a cały system bardziej skalowalny.

\subsubsection*{SignalR}

Użytkownik oprócz wysyłania pojedycznych żądań powinien móc śledzić stan swojego zamówienia. Jednak ciągłe odświeżadanie strony lub cykliczne wysyłanie zapytań do serwera o stan zamówienia nie jest najlepszym wyborem. Dlatego zdecydowano się na SignalR, technologię która najczęściej wykorzystuje protokół WebSocket. 

WebSocket zapewnia ciągłą, dwukierunkową komunikację przez jedno, stałe połączenie TCP. Aby nawiązać połączenie WebSocket aplikacja kliencka wysyła żądanie HTTP, co ma na celu uzgodnić protokół WebSocket. Serwer, który obsługuje WebSocket, akceptuje tą zmianę i nawiązuje stałe połączenie TCP. SignalR jest technologią, która wykorzystując najczęściej WebSocket, ułatwiając implementację dzięki dodatkowej warstwie abstrakcji.

Dzięki zapewnieniu stałego połączenia TCP, aplikacja internetowa w czasie rzeczywistym otrzymuje dane z aplikacji serwerowej o statusie zamówienia. Użytkownik może śledzić pozycję robota lub jakie produkty zebrał.

\subsection{Komunikacja aplikacji sterowania robotem i serwerowej}
\label{sec:komunikacjaRobotBackend}

Dwukierunkowa komunikacja w czasie rzeczywistym była konieczna dla aplikacji sterowania robotem i aplikacji serwerowej. Robot musiał wysyłać informacje o swoim stanie, to gdzie się znajduje i jaki ruch wykonał, aby użytkownik mógł zweryfikować status swojego zamówienia. Do tego zadania wykorzystano protokół MQTT.

MQTT to lekki protokół komunikacyjny oparty o wzorzec publish/subscribe. Znajduje zastosowanie w sieciach o niskiej przepustowości z uwagi na mały rozmiar nagłówków wiadomości. Charakteryzuje się niskim zużyciem energii przez co jest używany w urządzeniach zasilanymi bateriami. W protokole MQTT rozróżniamy dwa elementy: broker oraz klient. Broker MQTT jest centralnym serwerem zarządzającym komunikacją, z którym łączą się klienci aby za jego pomocą wysyłać dane. Klienci po połączeniu z brokerem wiadomości, subskrybują temat lub publikują wiadomości na dany temat. Jeśli klient wyśle wiadomość na dany temat to każdy klient, który subskrybuje ten sam temat otrzyma tą wiadomość.
